## Цель проекта

Всё началось как хобби одного студента по реверс-инжинирингу API LMS ЦУ
(потому что ему нравится ревёрсить в свободное время).

Со временем стало ясно, что многие студенты ЦУ недовольны качеством официальной
ЛМС, ибо она *работает далеко не так хорошо, как могла бы*. Кроме того,
сообщество хотело бы иметь инструменты, которые могли бы улучшить их взаимодействие
с LMS и адаптировать его под свои нужды.

Были изучены существующие попытки сделать это, после изучения которых стало
понятно, что они уж точно не подходят на роль надёжной основы продакшн-уровня.
Большинство сильно опирались на логику, сгенерированную с помощью ИИ:
спецификации были сгенерированы без строгой типизации или каких либо комментариев,
а клиентский код в основном был недетерминированным. Из-за этого таким инструментам
сложно заслужить доверие сообщества и безопасно использоваться сообществом
университета.

Это не попытка осудить кого-либо, кто работает над open-source альтернативами.
Наоборот, эти усилия ценны и заслуживают благодарности. Они помогли сделать
проблему явной: ЛМС ЦУ является проприетарным ПО с закрытым API, а стабилизация
заревёршенных спецификаций сложна для энтузиастов-одиночек.

Именно поэтому появился проект Open CU Services: чтобы предоставить открытую
инициативу по стабилизации заревёршенных спецификаций сервисов ЦУ и
предоставить сообществу разработчиков из ЦУ надёжные SDK.

Разработчики смогут строить инструменты на основе SDK вместо
того, чтобы поодиночке ревёрсить API. Ответственность же за ревёрс берёт на
себя этот проект.

SDK кодогенерируются (не путать с ИИ-генерацией) из заревёршенных спецификаций.
Это позволяет проекту предоставлять детерминированные и надёжные SDK, которые
уже можно считать продакшн-уровнем.

Главным приоритетом проекта является стабильность, поэтому от клиентов SDK не
стоит ожидать поддержки всех доступных ручек API с самого начала.

## Версионирование

В проекте используются строгие правила версионирования и именования для
[релизов спецификаций и SDK](https://github.com/cu-devs-collective/open-cu-services-sdk/releases),
что позволяет получать воспроизводимые сборки и предсказуемые обновления.

### Версионирование спецификаций и SDK

Для версионирования спецификаций и SDK используется [SemVer](https://semver.org/spec/v2.0.0.html).
Каждая версия состоит из трёх частей: `MAJOR.MINOR.PATCH`, например `0.1.0`.

Стоит заметить, что версии спецификации и SDK связаны между собой, но не
обязательно совпадают: у них разные роли.

Изменения в `MAJOR` и `MINOR` отражают изменения в спецификации и синхронизированы
с версиями SDK:

<!-- markdownlint-disable MD013 -->
- `MAJOR` - глобальные изменения в спецификации, например переход к стабильному релизу
- `MINOR` - новые эндпоинты или значительные изменения существующих, включая возможные переименования
<!-- markdownlint-enable MD013 -->

`PATCH` отражает либо небольшие изменения в спецификации, либо в SDK, например:

<!-- markdownlint-disable MD013 -->
- исправления схем внутри существующих ручек API (в том числе когда API на бэкенде меняется)
- изменения в шаблонах SDK или процессе кодогенерации
- периодические обновления зависимостей SDK
<!-- markdownlint-enable MD013 -->

> [!WARNING]
> Пока спецификация ещё не стабильна, новые `PATCH`-версии иногда могут включать
> критические изменения (например, изменение имён компонентов).
> Мы стараемся избегать таких ситуаций, когда это возможно.

### Именование компонентов спецификации

Мы стараемся сохранять имена компонентов стабильными, чтобы кодосгенерированные
типы внутри SDK оставались такими же. Однако изменения всё же могут появиться в
`MINOR`-версиях, а в редких случаях и в `PATCH`-версиях.

### Критические изменения

Критические изменения, например переименование компонентов, всегда документируется
во [вкладке Releases](https://github.com/cu-devs-collective/open-cu-services-sdk/releases)
в соответствующей секции.

Вероятность критических изменений зависит от типа обновления версии:

- `MAJOR` - ожидаются критические изменения
- `MINOR` - критические изменения возможны, но нечасты
- `PATCH` - критические изменения редки

### Версионирование релизов

Спецификации и SDK выпускаются в виде тегов в соответствующих ветках.

#### Версионирование релизов спецификации

Спецификация выпускается в ветке `master` и получает тег вида
`spec/{SDK_ID}/v{VERSION}`, где `SDK_ID` - это SDK ID из раздела [Available SDKs](#доступные-sdk),
а `VERSION` - версия спецификации. Например: `spec/lmsapi/v0.1.0`.

> [!NOTE]
> При релизе спецификации затронутые кодогенерируемые SDK перечисляются в
> описании этого же релиза.
> Таким образом, отдельные GitHub-релизы для каждого SDK-тега не создаются.

#### Версионирование релизов языковых SDK

SDK выпускается в своей SDK-ветке вида `{LANG}/{SDK_ID}`, где `LANG` - язык
соответствующей реализации SDK. SDK получает тег вида `{LANG}/{SDK_ID}/v{VERSION}`,
где `VERSION` - версия SDK. Например: `golang/lmsapi/v0.1.0`.

Ожидаемые значения `LANG` для языков:

- Go: `golang`
- TypeScript: `typescript`
- Python: `python`
- Dart: `dart`

## Структура репозитория

Проект представляет собой монорепозиторий со следующей структурой:

```text
.github/       CI-пайплайны для GitHub и автоматизация репозитория
dart/          Кодосгенерированные SDK на Dart
examples/      Примеры использования SDK на разных языках
gen/           Скрипты кодогенерации SDK, шаблоны и фиксирование зависимостей
golang/        Кодосгенерированные SDK на Go
python/        Кодосгенерированные SDK на Python
spec/          Заревёршенные спецификации OpenAPI
tools/         Инструменты для кодогенерации и патчинга
typescript/    Кодосгенерированные SDK на TypeScript
```
